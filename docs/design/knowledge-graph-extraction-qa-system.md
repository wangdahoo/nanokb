# 基于知识图谱的个人知识库智能抽取与问答系统 技术方案

## 1. 项目背景与核心理念

本方案旨在解决海量非结构化文档（OKF 格式、PDF、代码等）的知识管理难题。

**核心理念**：遵循 Andrej Karpathy 提出的 "LLM Wiki" 思想，摒弃传统的"即时检索"（RAG），采用 **"知识预编译"** 范式。系统将原始文档视作源码，通过自动化流水线将其"编译"为结构化的知识图谱（节点为概念/实体，边为关系），并以此为基础实现可推理、可溯源的智能问答。

## 2. 总体架构概览

系统采用五阶段流水线架构，分为"编译期"与"问答期"：

```mermaid
flowchart LR
    A["原始文档 raw/"] --> B["阶段一：增量检测与加载"]
    B --> C["阶段二：双轨知识抽取"]
    C --> D["阶段三：图谱编译与融合"]
    D --> E["阶段四：社区发现与索引"]
    E --> F["知识图谱 + 向量库"]
    F --> G["阶段五：问答推理引擎"]
    G --> H["用户交互 & 主动学习"]
```

## 3. 详细模块设计（编译期）

### 3.1 阶段一：增量检测与加载

- **机制**：使用 `watchdog` 监听 `raw/` 目录，计算文件 SHA256 哈希值，仅处理新增或变更的文件。
- **解析器**：集成 `unstructured` 库，支持 PDF、DOCX、Markdown、TXT 等格式的文本抽取。

### 3.2 阶段二：双轨知识抽取（核心）

根据文件类型分流处理，借鉴 Graphify 的设计哲学：

**代码轨（确定性抽取）**：

- 工具：`tree-sitter`（支持 Python/Java/JS 等）。
- 输出：提取函数签名、类定义、调用关系，零 Token 消耗，输出结构化 JSON。

**语义轨（LLM 抽取）**：

- 模型：GPT-4 / Claude 3.5 Sonnet（或本地 Ollama）。
- 策略：设计专门 Prompt，抽取 **概念(Concept)**、**实体(Entity)** 及 **关系(Relation)**。
- 输出示例：

```json
{
  "head": "Transformer",
  "relation": "relies_on",
  "tail": "Attention",
  "confidence": "EXTRACTED"
}
```

### 3.3 阶段三：图谱编译

- **存储**：内存中使用 NetworkX 构建 `MultiDiGraph`（有向多重图），大规模落地推荐 Neo4j。
- **元数据标注**：每条边和节点携带 `source_file`（来源）和 `confidence` 标签：
  - `EXTRACTED`（事实）：直接来源于原文。
  - `INFERRED`（推理）：LLM 逻辑推导，置信度中等。
  - `AMBIGUOUS`（歧义）：信息冲突，需人工审核。

### 3.4 阶段四：社区发现与索引

- **社区聚类**：应用 Leiden 算法对图谱进行无监督社区检测，自动归纳出"知识主题域"（如"深度学习"、"财务指标"），为宏观检索提供背景摘要。
- **双重索引**：
  - 关键词索引（whoosh 或 PostgreSQL tsvector）。
  - 向量索引（pgvector 或 ChromaDB），存储节点描述的 Embedding。

## 4. 详细模块设计（问答期）

基于构建好的图谱，问答流程采用 **"子图检索 + 提示编译"** 的混合策略，共 5 步：

### 4.1 查询理解

输入用户问题 Q。LLM 进行意图分类（事实/关系/总结）和实体识别（NER），提取关键节点名。

### 4.2 多路召回（三路并行）

1. **图谱检索（精确）**：以识别实体为中心，执行 2~3 跳子图扩展，获取结构化三元组。
2. **向量检索（模糊）**：将 Q 向量化，检索语义相似的节点描述或原始文本块。
3. **社区检索（宏观）**：定位实体所属的 Leiden 社区，获取该主题域的背景综述。

### 4.3 融合与重排序

合并三路结果，按 **置信度权重**（EXTRACTED > INFERRED）和相似度分数排序。裁剪冗余信息，确保 Prompt 上下文不超过模型窗口限制。

### 4.4 上下文编译

将筛选出的子图编译为 LLM 可读的纯文本形式，例如：

```text
[Transformer] --[relies_on]--> [Attention Mechanism] (来源: paper.md)
```

附带 GBrain 风格的时间线属性（如有），增强时效性理解。

### 4.5 生成与溯源

模型生成答案，强制要求标注引用 `^[source_file]`。若使用了 AMBIGUOUS 或 INFERRED 关系，附加提示："此结论为 AI 推理，建议核实源文件"。

## 5. 核心技术选型建议

| 功能模块 | 技术选型 | 说明 |
|---------|---------|------|
| 编程语言 | Python 3.10+ | 生态丰富，AI 支持完善 |
| 图谱处理 | NetworkX（开发）/ Neo4j（生产） | 开发期轻量，生产期专库专用 |
| 向量存储 | ChromaDB（本地）/ pgvector（生产） | 混合检索支持好 |
| LLM API | OpenAI / Anthropic / 本地 Ollama | 注重 Prompt 缓存以节省成本 |
| 代码解析 | Tree-sitter | 确定性抽取，零幻觉 |
| 任务编排 | Prefect / 自建脚本 | 支持断点续传与失败重试 |

## 6. 实施路线图（MVP 版本）

为避免过度设计，建议按以下迭代顺序落地：

- **第一阶段（骨架）**：建立 `raw/` 和 `out/` 目录，实现基于 `unstructured` 的文档加载器，支持 Markdown 和 TXT。
- **第二阶段（核心抽取）**：对接 OpenAI API，实现语义轨的 LLM 抽取（输出 JSON），存入 NetworkX。
- **第三阶段（基础问答）**：实现最简单的实体检索 + 2 跳子图查询，编译 Prompt 输出答案。
- **第四阶段（高级增强）**：加入 tree-sitter 代码解析、增量更新哈希检测、Leiden 社区聚类。
- **第五阶段（闭环优化）**：集成"主动学习"机制，将无法回答或冲突问题写入 `review_queue.md`，指导人工补充数据。

## 7. 总结

本方案将文档转化为 **"可计算的知识"**。通过编译期严格的实体关系抽取与问答期的多路混合召回，既保证了回答的事实准确性（来源追溯），又具备了超越单文档的逻辑推理能力（多跳关系）。这不仅是文档检索系统，更是一个会随着数据增长而不断进化的"数字第二大脑"。
